## 1. Codex 失败根因：verifier 脚本 CRLF，agent 工作成果报废

**Codex 本身成功完成了两个任务的编码**（正常走完 agent 阶段并输出 trajectory）。失败全部发生在 verifier：

- `jobs/run-000003-codex/*/verifier/test-stdout.txt` 仅一行：
  `bash: line 1: /tests/test.sh: cannot execute: required file not found`
- 这是 shebang 变成 `#!/bin/bash\r` 的典型症状。实测确认 `tasks/happy-dom-abort-pending-body-reads/tests/test.sh` 与 `tasks/narwhals-rolling-window-suite/tests/test.sh` 在磁盘上均为 **CRLF 行尾**
- 链路：test.sh 无法执行 → 无 reward 文件 → `RewardFileNotFoundError`（pier 明确不重试该异常）→ 两个 trial 全失败 → pier 退出码 1 → Run 判 failed

根因：`git config core.autocrlf=true` 在 checkout 时把 `.sh` 转成了 CRLF。已添加的 `.gitattributes`（`*.sh text eol=lf`）**只约束未来的 checkout，不会重写工作区现有文件**，因此本次运行时磁盘文件仍是 CRLF。

该问题对三个 Agent 一视同仁：claude-code 的 narwhals trial 的 verifier 报了同样错误。**即使另外两个 Agent 全部跑完，本次运行也注定全军覆没**——这是"钱花了、分拿不到"的最坏失败模式。

## 2. claude-code / mini-swe-agent 迟迟无结果的原因

两者最终都在 08:01:2x 被外部终止：job 级 `result.json` 的 `finished_at` 为 `null`、trial 仍处 running、docker-cleanup 审计（trigger=`run-finished`）显示强制移除了仍在运行的容器，说明 pier 并非自然结束。若当时是通过 UI"取消"按钮停止，状态本应记 `cancelled` 而非 `failed`（+"Pier 进程退出码 1"），此差异指向进程是被直接杀掉的；如确认点的是取消按钮，则取消状态标注存在竞态，需另行排查。

### mini-swe-agent：慢但正常，"还在干活"而非卡死

- 严格串行的单步 ReAct agent，`xhigh` 推理使每步都慢；
- narwhals trial 在容器内跑全量 pytest（日志可见 `10134 passed, 1237 skipped`），单轮验证十几分钟起；
- happy-dom trial 已完成实质修改（5 个文件，+395/−180），在执行 `git commit` 时撞上容器无 git 身份的报错（`fatal: unable to auto-detect email address`）；该报错在日志中仅出现 1 次，紧接着进程就被 08:01 的终止杀掉——它是"死在自救之前"，而非反复空转（经查证这一环境状态是评测设计的一部分，见 §2.1）；
- agent timeout 为 5400s（90 分钟），被终止时仅运行约 50 分钟，尚未超时。

#### §2.1 补充查证：容器无 git 身份是评测设计，不是缺陷

- "IMPORTANT: Please work on this in a new branch from main and commit everything" 来自每个任务自带的 `instruction.md`（113 个任务统一模板），属 DeepSWE 任务集本身的要求；
- 评分管道刻意以"已提交的工作"为准：任务的 `pre_artifacts.sh` 用 `git diff <起始commit> HEAD` 生成 `model.patch`（注释明确写 "Capture the agent's **committed** work"）。**commit 失败 → HEAD 不动 → patch 为 0 字节 → verifier 应用空补丁 → reward 必为 0**，这是硬性约束；
- 但该障碍是可自救的：本次 codex 在两个 trial 中都撞上了同样问题，随即执行 `git config --global user.email/user.name` 后成功提交（model.patch 分别 25KB / 42KB）。官方统计（164 次 trial）中 `happy-dom-abort-pending-body-reads` 通过率 91.46%、`narwhals-rolling-window-suite` 87.2%，证明官方跑法同样处于无身份环境且普遍可越过；
- 结论：**不应在镜像或环境里预置 git 身份**——官方 pass_rate 就是在这个环境下校准的，预置会让本地测量偏离官方口径（变相放水）。处理不完美环境本身是任务的隐含考察点。

### claude-code：一个 trial 秒死，另一个陷入失控循环

- narwhals trial（07:13–07:17）：启动即连续 503（经 pier egress 代理返回，上游网关过载），Claude Code 内建重试 10 次全部失败后以 `api_error` 退出；pier 基础设施重试 1 次后仍失败；
- happy-dom trial：见第 3 节的 subagent 风暴，43 分钟不收敛直至被终止；
- 额外发现 pier 在 Windows 上的编码缺陷：`Failed to convert Claude Code events to trajectory: 'gbk' codec can't decode byte 0x94...`——pier 以系统默认 GBK 读取 UTF-8 事件文件。即使运行正常结束，trajectory/token 统计也可能解析失败；
- 数据库中这两个 Run 的 token/费用为 `null`：pier 被杀时 job 汇总未写完，UI 无数据可展示，因此界面上"看不到结果"。

## 3. 天量请求根因：容器内 Claude Code 的 subagent 风暴

请求量尖峰主体来自 claude-code 的 happy-dom trial（数据统计自该 trial 278 MB 的 `claude-code.txt` 与 125 MB 的 sessions 目录）：

| 指标 | 数值 |
|---|---|
| 主 agent API 请求 | 36 次 |
| 主 agent 发起的 Agent 工具调用 | **2026 次**（平均每轮约 56 个） |
| 实际产生的 subagent 会话 | 2687 个 |
| subagent 发出的 API 请求 | **4058 次**（07:18–08:01，≈1.6 请求/秒 持续 43 分钟） |
| 总输入 token | **66,630,573**（其中未缓存 29,064,621） |
| 总输出 token | 2,026,289 |

失控模式：`gpt-5.6-sol` 在 Claude Code harness 中写完代码后进入"自我审查循环"——反复以相同角度批量 spawn 审查 subagent（`Audit removed behavior`×155、`Scan diff linewise`×139、`Check repository conventions`×108、`Wait for review agents`×84……），永不满足收敛条件。**`--max-turns 80` 只限制主对话轮数，对 subagent 的数量与消耗完全无约束**，且每个 subagent 都是携带独立上下文的完整 agent。

按本项目估价公式（未缓存输入 $5/M、缓存 $0.5/M、输出 $30/M；该价目与 codex 的 pier 报告值 $10.578 精确吻合），该 trial 的实际成本约为：

> 29.06M × $5 + 37.57M × $0.5 + 2.03M × $30 ≈ **$225**（未入库，因 pier 被终止且遭遇 GBK 解析缺陷）

本次试运行三个 Run 合计实际消耗约 **$235+**（codex $10.578 + claude-code ≈$225 + mini 未统计）。

叠加因素：

1. `xhigh` effort 是全局放大器：codex 两个 trial 各消耗约 720 万输入 token（97%+ 为缓存命中，但按网关计价仍记 $10.578）；
2. 6 个 agent 容器并发打同一网关，网关过载后引发 503；503 又触发"Claude Code 内建 10 次重试 × pier 基础设施重试 1 次"的重试放大。

## 4. 规避措施（按优先级）

1. **先修 verifier，再花一分钱**：批量将任务脚本转回 LF 并重新规范化 git 索引；preflight 增加零成本检查——校验 `tests/test.sh` 无 CRLF、shebang 可执行。杜绝"agent 成功、verifier 必挂"的白烧模式。
2. **禁用容器内 Claude Code 的 subagent**：`--disallowedTools` 追加 `Task`（当前仅禁 `EnterPlanMode`）。基准测试需要单 agent 可比口径，此一项可消除本次 90% 以上的用量。
3. **运行中成本熔断**：为 Run 增加 token/费用上限，运行中周期核算（含进行中 trial 的增量用量），超限自动走既有取消+清理链路。同时 mini-swe-agent 的 `agent.cost_limit=0`（无限制）应替换为真实上限/步数上限。
4. **网关侧兜底**（本仓库之外）：对该 key 设每日消费与并发配额，过载时排队而非 503；消费告警接通知渠道，而非事后看图表。
5. **试运行降档**：冒烟用 low/medium effort + 单任务单 trial，端到端确认（含 verifier 产出 reward）后再上 xhigh/全量。
6. **修复 pier 的 GBK 读取缺陷**（Windows 下强制 UTF-8），否则正常运行也可能拿不到 token/费用统计。
7. ~~容器内 git 身份配置~~：经查证（§2.1）撤回该建议——无 git 身份是评测环境设计的一部分，预置身份会偏离官方测量口径。

## 5. 修复记录（2026-07-12，与本报告同批交付）

对应第 4 节第 1/2/3/6 项；网关侧配额（第 4 项）与运行降档（第 5 项）属运维/操作约定，不在代码修复范围内。验证：后端 pytest 48 项全部通过（含 5 项新增护栏测试）、前端 `tsc --noEmit` 与 Vite 构建通过、TestClient 冒烟通过。

1. **任务脚本行尾修复**：`tasks/**` 下 338 个 CRLF 的 `.sh` 全部转为 LF 并 `git add --renormalize`。git 对象库中本来就是 LF（是 `core.autocrlf=true` 在 checkout 时污染了工作区），配合已有 `.gitattributes`（`*.sh text eol=lf`）后不会再复发。
2. **preflight 拦截 CRLF**（`backend/app/runner.py` `_preflight`）：启动前扫描所选任务的全部 `*.sh`，发现 CRLF 直接拒绝运行并指明文件，杜绝"agent 成功、verifier 必挂"的白烧模式。
3. **禁用容器内 Claude Code 的 subagent**（runner claude-code 分支）：`--agent-kwarg disallowed_tools=EnterPlanMode,Task,Agent`（覆盖 pier 默认值并追加 Task/Agent），消除本次事故 90% 以上的用量来源。
4. **mini-swe-agent 步数护栏**（runner mini 分支）：经 `config_file` 注入 `agent.step_limit: 250`。其原生 `cost_limit` 依赖 litellm 价格表，对自建网关模型恒算 0 不可用，步数上限是唯一确定性护栏。
5. **运行中用量熔断**（runner `_wait_with_guard`，每 20 秒核查一次）：
   - 费用线：已落盘 Trial 的累计费用（pier 报告值优先、缺失按 token 估算）达到 `run_budget_usd`（新增设置项，默认 $10，0 禁用，Settings 页可改）即终止；
   - 失控线（硬保护，不可关）：单 Trial agent 日志 ≥ 30MB，或 subagent 会话文件 ≥ 60 个，即判定失控终止。以事故 trial 的写入速率（6.5MB/分钟、62 个会话/分钟）推算，本次损失会在启动后约 1–5 分钟内被拦停，约 $2 以内；
   - 触发后走既有终止+取消状态+Docker 清理链路，Run 标记 `cancelled` 并写明"用量护栏自动终止"原因。
6. **pier GBK 缺陷修补**：本机 pier 安装的 `agents/installed/claude_code.py` 读取会话文件改为强制 `encoding="utf-8"`（遵循项目已有的 pier 本地补丁惯例）；Diagnostics 新增 `pier-claude-utf8` 检查项，pier 升级丢失补丁时会在预检中报错。

**残留事项**：xhigh effort 的放大效应与网关侧配额告警需靠操作约定与网关配置解决。容器内 git 无身份配置经查证（§2.1）不是缺陷、无需修复。另外，若 08:01 的终止操作当时走的是 UI 取消按钮，则状态被记为 `failed`（而非 `cancelled`）的路径值得复查——本轮证据（`trigger=run-finished`）更支持进程是被直接杀掉的。
